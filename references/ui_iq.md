# UI 设计

## CS vs BS 分析与决策

| 维度     | CS（桌面客户端）                 | BS（浏览器端）                     |
|----------|----------------------------------|------------------------------------|
| 开发难度 | 高（Electron/Tauri/PyQt 等）     | 低（HTML/CSS/JS，迭代快）          |
| 部署安装 | 需安装/更新包                    | 免安装，本地打开或本地服务即可     |
| 系统能力 | 可深度调用 OS/硬件               | 受限于浏览器沙箱（本地需配合后端） |
| 多端一致 | 需分平台构建                     | 同一套页面多端可用                 |
| 与状态联动 | 可直读本地 runtime/state/日志          | 需通过 file:// 或本地接口读 JSON   |

**决策：采用 BS（浏览器端）。**  
理由：本技能以「状态展示 + 行为历史」为主，不依赖复杂系统能力；BS 开发与迭代成本低、易自进化，且当前骨架已为 HTML，可与 `runtime/state/current_mission.json` 及后续本地 API 对接。若未来需桌面级能力（如全局快捷键、托盘），再考虑用本地小服务 + BS 前端或封装为 CS。

## 科幻主体

- **风格**：类似贾维斯/星期五的科幻主体，持续自我完善。
- **载体**：BS，前端骨架在 `assets/`，可随进化迭代。
- **悬浮球**：PyQt5 版仅展示「FRIDAY」标题、当前阶段、使命、轮次；中部为轮次，底部「双击 · 查看过程」打开弹框展示行为日志（来自 `runtime/state/recent_logs.json` 或 runtime/logs/）。**不再用年龄或智商阶段**——思考能力由当前驱动本技能的 LLM 决定，不同模型能力不同。

## 与状态的关系

- UI 可读取 `runtime/state/current_mission.json` 展示「当前在干什么」。
- 日志可被 UI 展示为「行为历史」，便于溯源。
